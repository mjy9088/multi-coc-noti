# 자원 상태 기반 업그레이드 알림 정책

## 목적

모든 업그레이드에 고정된 `60분 전, 1분 전, 완료 시` 알림을 보내는 대신, 사용자가 마을 단위로 대략 입력한 자원 여유 상태에 따라 실제 행동이 필요한 시점만 알린다.

자원 상태는 정밀한 자원량이나 업그레이드별 값이 아니다. 사용자가 게임 export를 반영할 때 다음 업그레이드를 시작할 준비가 어느 정도인지 빠르게 답하는 마을 단위 상태다.

## 자원 상태

DB에는 다음 네 상태를 서로 구분해 저장한다.

| 상태 | 의미 | 알림 계산 |
| --- | --- | --- |
| `abundant` | 자원 많이 남음 | 완료 시 알림만 |
| `sufficient` | 충분함 | 완료 1분 전 알림만 |
| `insufficient` | 부족함 | 자원 준비 알림 + 완료 시 알림 |
| `unanswered` | 팝업에서 응답하지 않음 | DB에는 별도 저장하지만 `insufficient`처럼 계산 |

상태는 마을 전체에 하나만 저장한다. 여러 업그레이드가 진행 중이어도 업그레이드별로 질문하거나 별도 상태를 저장하지 않는다.

## 자원 준비 예상 시간

마을마다 분 단위의 `resourcePreparationMinutes`를 설정한다.

- 양의 정수: 자원 준비 알림 사용
- `null`: 자원 준비 알림 끄기
- 상태가 `insufficient` 또는 `unanswered`일 때만 사용
- 준비 알림을 꺼도 부족 상태의 완료 알림은 유지

설정 UI는 `Settings → Manage villages`에 토글과 분 단위 입력으로 제공한다. 자원 상태도 같은 화면에서 언제든 수정할 수 있다.

초기 마이그레이션 값은 다음과 같다.

- 기존·신규 마을의 상태: `unanswered`
- 자원 준비 예상 시간: `60분`, 켜짐

따라서 사용자가 새 정책에 아직 응답하지 않은 마을은 기존 60분 알림과 유사한 자원 준비 알림 및 완료 알림을 받는다. 기존 1분 전 알림은 `sufficient` 상태에서만 사용한다.

## 게임 export 업데이트 흐름

진행 중인 업그레이드가 포함된 게임 export를 반영할 때마다 자원 상태를 묻는다.

1. JSON을 붙여넣거나 `Quick Paste`를 사용한다.
2. 서버 미리보기로 마을과 업그레이드를 확인한다.
3. 사용자가 Import를 누르면 export를 먼저 `unanswered` 상태로 저장하고 알림을 계산한다.
4. 저장이 완료되면 자원 상태 팝업을 연다.
5. `자원 많이 남음`, `충분함`, `부족함` 중 하나를 선택하면 상태를 별도로 저장하고 미래 알림을 다시 계산한다.
6. 닫기, 바깥 영역 선택, `나중에 답하기`로 응답하지 않으면 이미 저장된 `unanswered` 상태를 유지한다.

진행 중인 업그레이드가 없는 export에는 팝업을 표시하지 않는다. 자동 Push/Pull snapshot 수집도 팝업을 표시할 수 없으므로 마지막으로 저장된 마을 상태를 유지한다.

팝업은 export 저장을 취소시키는 확인창이 아니다. 팝업이 열릴 때는 export가 이미 반영된 상태이며, 응답 없이 닫아도 `unanswered` 상태를 유지한다.

## 알림 스케줄

<!-- contract: ALERT-PLAN-001 -->

각 활성 업그레이드는 마을의 현재 자원 상태로 다음 예약을 만든다.

### 자원 많이 남음

```text
완료 시: 업그레이드 완료 알림
```

### 충분함

```text
완료 1분 전: 업그레이드가 곧 완료된다는 알림
```

완료 시 알림은 추가로 보내지 않는다.

### 부족함 또는 무응답

준비 알림을 켠 경우:

```text
완료 시각 - 자원 준비 예상 시간: "자원을 미리 준비하세요!"
완료 시: 업그레이드 완료 알림
```

준비 알림을 끈 경우:

```text
완료 시: 업그레이드 완료 알림
```

## 즉시 준비 알림

export 반영 또는 설정 변경 시점에 업그레이드 완료까지 남은 시간이 자원 준비 예상 시간보다 짧거나 같으면 준비 알림을 즉시 발송 가능한 상태로 예약한다.

예시:

- 완료까지 40분
- 자원 준비 예상 시간 60분
- 상태 `insufficient`

이 경우 준비 알림의 원래 시각이 이미 지났으므로 다음 Notifier 처리 주기에 즉시 발송한다. HTTP 요청 처리 중 Bark를 직접 호출하지 않고 DB 큐를 사용한다.

## 자원 준비 알림 중복 방지

`자원을 미리 준비하세요!` 알림은 업그레이드가 아니라 마을 단위로 억제한다.

- 중복 범위: 같은 마을 + `resource_preparation` 알림 유형
- 억제 기간: 마지막으로 성공한 준비 알림 시각부터 그 알림에 사용한 준비 예상 시간
- 억제 기간 안에 여러 업그레이드의 준비 알림이 도래해도 하나만 발송
- 다른 마을의 준비 알림에는 영향 없음
- 완료 알림은 억제 대상이 아니며 부족 상태에서는 업그레이드마다 발송

동시에 여러 준비 알림이 처리되는 경우에도 하나만 발송하도록 DB에서 원자적으로 선점한다. 실패한 Bark 요청은 성공으로 보지 않으므로 기존 재시도 정책을 따른다.

예시:

```text
10:00 마을 A의 준비 알림 발송, 준비 시간 45분
10:20 마을 A의 다른 업그레이드 준비 알림 → 억제
10:46 마을 A의 다른 업그레이드 준비 알림 → 발송 가능
10:20 마을 B의 준비 알림 → 별개이므로 발송 가능
```

## 상태·설정 수정

자원 상태와 준비 시간은 `Manage villages`에서 언제든 변경할 수 있다.

- 변경 시 아직 발송하지 않은 해당 마을의 알림을 새 정책으로 다시 계산
- 이미 성공한 알림 기록은 삭제하거나 되돌리지 않음
- 준비 시간을 끄면 아직 발송하지 않은 준비 알림을 취소
- `sufficient`로 바꾸면 완료 1분 전 알림만 유지
- `abundant`로 바꾸면 완료 알림만 유지
- `insufficient` 또는 `unanswered`로 바꾸면 준비 설정에 따라 준비 알림을 만들고 완료 알림을 유지
- 변경 시 이미 준비 시각이 지났다면 중복 억제 조건을 확인한 뒤 즉시 예약

## 데이터 모델

마을 계정에는 다음 값을 추가한다.

```text
resource_status: abundant | sufficient | insufficient | unanswered
resource_status_updated_at: timestamptz
resource_preparation_minutes: integer | null
```

알림 예약은 목적을 구분하는 유형을 저장한다.

```text
notification_kind: completion | one_minute | resource_preparation
```

자원 준비 알림은 특정 업그레이드에서 계산되더라도 중복 억제 키는 마을 ID와 `notification_kind`를 사용한다. 발송 기록에는 당시 준비 시간도 보존해 이후 설정 변경이 이미 시작된 억제 기간을 바꾸지 않게 한다.

## 기존 알림 설정 전환

새 정책은 업그레이드별 `notificationOffsets`와 `60, 1, 0` 직접 입력 UI를 대체했다.

- 마이그레이션 후 활성 업그레이드의 미발송 예약을 마을 자원 상태로 다시 생성
- 이미 `sent`인 기록은 감사·중복 방지를 위해 유지
- 처리 중이거나 재시도 대기 중인 기존 고정 offset 알림은 새 정책에 없는 경우 취소
- 히스토리 백업은 마을 자원 상태와 준비 시간을 포함하도록 확장

## Bark 문구

한국어 예시:

- 자원 준비: `{마을 이름}: 자원을 미리 준비하세요! {업그레이드 이름} 완료까지 약 {시간}`
- 1분 전: `{마을 이름}: {업그레이드 이름} 완료까지 1분 남았습니다.`
- 완료: `{마을 이름}: {업그레이드 이름} 레벨 {다음 레벨} 완료`

영어 locale에도 같은 의미의 문구를 제공한다.

## 완료 조건

- 활성 업그레이드가 있는 export는 응답을 기다리지 않고 `unanswered`로 먼저 반영한 뒤 자원 상태 팝업을 표시한다.
- 무응답은 DB에서 구분되고 부족함과 같은 예약 결과를 만든다.
- 네 상태별 알림 시점이 정책 표와 일치한다.
- 준비 알림을 끄면 준비 알림만 사라지고 부족 상태의 완료 알림은 유지된다.
- 준비 시각이 이미 지난 경우 다음 Notifier 주기에 발송 가능하다.
- 같은 마을의 준비 알림은 준비 시간 구간 안에 중복 발송되지 않는다.
- 상태·준비 시간을 수정하면 미발송 예약만 안전하게 재계산된다.
